## 1. 시계열 예측의 기본 개념

### 1.1 왜 예측 모델이 필요한가

Part 1에서 "과거에 무슨 일이 있었는가"를 분석했다면, Part 2에서는 **"앞으로 어떻게 될 것인가"**를 예측한다. 예측 모델의 비즈니스 활용:

| 활용 영역 | 예시 |
|-----------|------|
| 재고 관리 | 다음 주 매출 예측 → 적정 재고 수준 결정 |
| 인력 배치 | 성수기/비수기 예측 → 인력 스케줄링 |
| 프로모션 계획 | 프로모션 효과 예측 → ROI 최적화 |
| 경영 계획 | 연간 매출 예측 → 예산 수립 |

### 1.2 시계열 데이터의 구성 요소

시계열 데이터는 네 가지 요소로 분해할 수 있다:

```
관측값 = 트렌드 + 계절성 + 주기성 + 잔차(노이즈)
```

| 요소 | 설명 | 예시 |
|------|------|------|
| **트렌드** | 장기적 상승/하락 방향 | 매년 매출이 5%씩 성장 |
| **계절성** | 일정 주기의 반복 패턴 | 매년 12월 매출 급증 |
| **주기성** | 비정기적 경기 변동 | 경제 호황/불황 |
| **잔차** | 설명할 수 없는 변동 | 예측 불가능한 이벤트 |

## 2. 피처 엔지니어링 (Feature Engineering)

### 2.1 피처 엔지니어링이란

원본 데이터에서 **모델이 학습하기 좋은 형태의 새로운 변수**를 만드는 과정이다. 모델 성능의 80%는 피처 엔지니어링에 의해 결정된다고 해도 과언이 아니다.

### 2.2 시계열 피처의 종류

| 피처 유형 | 예시 | 목적 |
|-----------|------|------|
| **시간 피처** | 요일, 월, 연도, 주차 | 계절성/주기성 포착 |
| **래그 피처** | 7일 전 매출, 30일 전 매출 | 자기상관(autocorrelation) 활용 |
| **롤링 피처** | 7일 이동평균, 30일 이동평균 | 단기 트렌드 포착 |
| **외부 피처** | 유가, 프로모션 수, 휴일 여부 | 외부 요인 반영 |

### 2.3 래그 피처와 데이터 누수 (Data Leakage)

래그 피처는 **과거 시점의 값을 현재 시점의 입력으로 사용**하는 것이다:
- `lag_7`: 7일 전 매출 → "지난주 같은 요일 매출이 이번 주를 예측하는 데 도움"
- `lag_30`: 30일 전 매출 → "지난달 같은 시점의 매출 참고"

**데이터 누수(Data Leakage)**란 모델이 **예측 시점에 알 수 없는 미래 정보**를 학습에 사용하는 것이다. 예를 들어:
- 오늘 매출을 예측하는데 내일 매출 데이터를 피처로 사용 → 데이터 누수
- 래그 피처는 과거 데이터만 사용하므로 누수가 아님

데이터 누수가 발생하면 학습 시 정확도는 높지만, 실제 운영에서는 성능이 급락한다.

> **더 알아보기**
> - [Data Leakage in Machine Learning — Kaggle](https://www.kaggle.com/code/alexisbcook/data-leakage)
> - [Feature Engineering for Time Series — Towards Data Science](https://towardsdatascience.com/feature-engineering-for-time-series-data-a-comprehensive-overview-e3eadc0e42f6)

## 3. 모델 선택과 비교

### 3.1 Prophet

Facebook(Meta)이 개발한 시계열 예측 모델. 비즈니스 시계열에 특화되어 있다:
- **장점**: 계절성/휴일 효과 자동 처리, 직관적 파라미터, 결측치에 강건
- **단점**: 외부 변수(유가, 프로모션) 반영이 제한적, 복잡한 비선형 관계 포착 어려움
- **적합한 상황**: 빠른 베이스라인 모델, 계절성이 강한 데이터

Prophet은 시계열을 트렌드 + 계절성 + 휴일 효과로 분해하는 **가법 모델(Additive Model)**이다.

> **더 알아보기**
> - [Prophet 공식 문서](https://facebook.github.io/prophet/)

### 3.2 XGBoost / LightGBM / CatBoost

**Gradient Boosting** 계열의 트리 기반 모델들이다:

| 모델 | 특징 |
|------|------|
| **XGBoost** | 가장 널리 사용, 정규화 내장, Kaggle 우승 다수 |
| **LightGBM** | 빠른 학습 속도, 대용량 데이터에 유리, leaf-wise 분할 |
| **CatBoost** | 범주형 변수 자동 처리, 순서형 부스팅으로 과적합 방지 |

이 모델들은 시계열의 시간 구조를 직접 이해하지 못한다. 대신 **피처 엔지니어링**으로 시간 정보(래그, 롤링, 요일 등)를 명시적으로 제공해야 한다.

> **더 알아보기**
> - [XGBoost 공식 문서](https://xgboost.readthedocs.io/)
> - [LightGBM 공식 문서](https://lightgbm.readthedocs.io/)
> - [CatBoost 공식 문서](https://catboost.ai/docs/)

### 3.3 모델 비교 기준

| 기준 | 설명 |
|------|------|
| **RMSE** | 예측 오차의 크기 (낮을수록 좋음) |
| **MAE** | 절대 오차 평균 (이상치에 덜 민감) |
| **MAPE** | 비율 기반 오차 (규모가 다른 매장 비교 가능) |
| **학습 시간** | 실무 운영 가능성 |
| **해석 가능성** | 비즈니스 의사결정자에게 설명 가능한가 |

## 4. 시계열 교차 검증 (TimeSeriesSplit)

### 4.1 왜 일반 교차 검증을 쓰면 안 되는가

일반적인 K-Fold 교차 검증은 데이터를 **무작위로 섞어서** 나눈다. 시계열에서 이렇게 하면:
- 2017년 데이터로 학습하고 2015년을 예측하는 상황 발생
- 미래 정보가 과거 예측에 사용됨 → **데이터 누수**

### 4.2 TimeSeriesSplit

시계열 교차 검증은 **시간 순서를 유지**하면서 데이터를 나눈다:

```
Fold 1: [Train: 2013] → [Test: 2014]
Fold 2: [Train: 2013-2014] → [Test: 2015]
Fold 3: [Train: 2013-2015] → [Test: 2016]
Fold 4: [Train: 2013-2016] → [Test: 2017]
```

각 fold에서 학습 데이터는 항상 테스트 데이터보다 **과거**에 위치한다. 이렇게 해야 실제 운영 환경(과거 데이터로 미래를 예측)과 동일한 조건에서 모델을 평가할 수 있다.

> **더 알아보기**
> - [scikit-learn TimeSeriesSplit 공식 문서](https://scikit-learn.org/stable/modules/generated/sklearn.model_selection.TimeSeriesSplit.html)

## 5. 하이퍼파라미터 튜닝 (Optuna)

### 5.1 하이퍼파라미터란

모델이 **스스로 학습하지 않는**, 사람이 설정해야 하는 값이다:
- 트리 깊이 (`max_depth`): 너무 깊으면 과적합, 너무 얕으면 과소적합
- 학습률 (`learning_rate`): 너무 크면 수렴 불안정, 너무 작으면 학습 느림
- 트리 수 (`n_estimators`): 많을수록 정확하지만 과적합 위험

### 5.2 Optuna의 작동 원리

Optuna는 **베이지안 최적화(Bayesian Optimization)** 기반 자동 튜닝 프레임워크다:

1. 초기에 몇 가지 파라미터 조합을 **랜덤 탐색**
2. 결과를 바탕으로 **유망한 영역을 추정** (TPE 알고리즘)
3. 유망한 영역에서 **집중 탐색**
4. 반복하여 최적 조합 발견

Grid Search(모든 조합 시도)보다 훨씬 효율적이다 — 100개 조합 중 20~30개만 시도해도 비슷한 결과를 얻을 수 있다.

> **더 알아보기**
> - [Optuna 공식 문서](https://optuna.readthedocs.io/)
> - [Optuna 튜토리얼 (Medium)](https://medium.com/optuna/using-optuna-to-optimize-xgboost-hyperparameters-63bfcdfd3407)

## 6. 모델 해석 (SHAP)

### 6.1 왜 모델 해석이 필요한가

예측 모델이 "다음 달 매출이 10% 증가할 것"이라고 했을 때, 의사결정자는 반드시 묻는다: **"왜?"**

모델 해석 없이는:
- 모델을 신뢰할 수 없음
- 예측이 틀렸을 때 원인을 파악할 수 없음
- 비즈니스 액션으로 연결할 수 없음

### 6.2 SHAP (SHapley Additive exPlanations)

SHAP은 게임 이론의 **Shapley Value**를 기반으로 각 피처의 기여도를 계산한다:

핵심 아이디어: "이 피처가 없었다면 예측값이 얼마나 달라졌을까?"

```
예측값 = 기본값(평균) + 피처1 기여도 + 피처2 기여도 + ... + 피처N 기여도
```

SHAP의 주요 시각화:
- **Summary Plot**: 전체 피처 중요도 + 각 피처 값의 방향성
- **Waterfall Plot**: 개별 예측에 대한 피처별 기여도 분해
- **Dependence Plot**: 특정 피처 값에 따른 SHAP 값 변화

### 6.3 SHAP 해석 예시

"프로모션 수(onpromotion)의 SHAP 값이 +500"이라면:
- 이 매장의 프로모션 수가 매출 예측을 **500만큼 높이는 방향**으로 기여했다
- 프로모션이 많을수록 매출 예측이 높아진다

> **더 알아보기**

## 7. 시나리오 분석

### 7.1 시나리오 분석이란

예측 모델에 **"만약 ~한다면?"** 질문을 던지는 것이다:
- "프로모션을 20% 늘리면 매출이 얼마나 오를까?"
- "유가가 $80로 상승하면 매출에 어떤 영향이 있을까?"
- "새 매장을 열면 기대 매출은?"

### 7.2 시나리오 분석의 구조

```
현재 상태 (Baseline) → 변수 변경 (What-if) → 예측 변화 (Impact)
```

1. **Baseline 설정**: 현재 조건에서의 예측값
2. **변수 변경**: 분석하고 싶은 변수만 변경 (나머지는 고정)
3. **Impact 계산**: 변경 후 예측값 - Baseline 예측값

### 7.3 프로모션 ROI 최적화

시나리오 분석의 대표적 활용은 **프로모션 ROI 최적화**다:

```
프로모션 ROI = (프로모션으로 인한 매출 증가분) / (프로모션 비용)
```

모든 카테고리에 동일한 프로모션을 적용하는 것보다, **SHAP 기여도가 높은 카테고리에 집중**하는 것이 효율적이다. 시나리오 분석으로 최적의 프로모션 배분을 찾을 수 있다.

## 8. 실무 적용 시 주의사항

### 8.1 모델은 과거의 패턴을 학습한다

예측 모델은 **과거 데이터에 존재하는 패턴**만 학습할 수 있다. 따라서:
- 코로나19 같은 전례 없는 이벤트는 예측 불가
- 시장 구조가 변하면 모델 재학습 필요
- 정기적인 모델 모니터링과 재학습 파이프라인이 필수

### 8.2 예측 불확실성

모든 예측에는 불확실성이 있다. 단일 숫자("매출 1억")보다 **구간 예측**("매출 9000만~1.1억, 95% 신뢰구간")이 더 정직하고 유용하다. Prophet은 기본적으로 신뢰구간을 제공한다.

### 8.3 모델 vs 비즈니스 판단

모델은 의사결정을 **보조**하는 도구이지, 대체하는 것이 아니다. 모델이 "프로모션을 늘려라"고 해도, 예산 제약, 브랜드 전략, 경쟁사 동향 등 모델이 알 수 없는 요소를 고려해야 한다.
